# STL 时间复杂度和底层实现总结

STL 容器面试时通常会围绕三个问题展开：

- 底层数据结构是什么
- 常见操作的时间复杂度是多少
- 什么时候会触发迭代器、指针、引用失效

量化交易系统里还会继续追问性能细节：内存是否连续、cache locality 是否友好、是否会动态分配、是否可能 rehash 或扩容。

## 1. 顺序容器

| 容器 | 底层实现 | 随机访问 | 头部插入/删除 | 尾部插入/删除 | 中间插入/删除 | 典型特点 |
| --- | --- | --- | --- | --- | --- | --- |
| `vector` | 连续动态数组 | `O(1)` | `O(n)` | 均摊 `O(1)` | `O(n)` | 内存连续，cache 友好，可能扩容 |
| `deque` | 分段连续数组 | `O(1)` | `O(1)` | `O(1)` | `O(n)` | 头尾操作快，但整体不完全连续 |
| `list` | 双向链表 | `O(n)` | `O(1)` | `O(1)` | 已知位置 `O(1)` | 节点分散，cache 不友好 |
| `forward_list` | 单向链表 | `O(n)` | `O(1)` | `O(n)` | 已知前驱 `O(1)` | 更省节点指针空间，只能单向遍历 |
| `array` | 固定大小数组 | `O(1)` | 不支持动态插入 | 不支持动态插入 | 不支持动态插入 | 大小编译期固定，零额外动态分配 |

### 1.1 `vector`

`vector` 是动态数组，底层是一段连续内存。常见实现中会维护三个指针：

- `start`：指向首元素
- `finish`：指向最后一个元素之后的位置
- `end_of_storage`：指向已分配空间的末尾

核心复杂度：

| 操作 | 复杂度 | 说明 |
| --- | --- | --- |
| `operator[]` / `at()` | `O(1)` | 直接根据偏移访问 |
| `push_back()` / `emplace_back()` | 均摊 `O(1)` | 扩容时需要搬迁全部元素 |
| `insert(pos, x)` | `O(n)` | pos 之后元素需要整体后移 |
| `erase(pos)` | `O(n)` | pos 之后元素需要整体前移 |
| `find()` | `O(n)` | 未排序时只能线性查找 |
| `clear()` | `O(n)` | 析构所有元素，容量通常不变 |

迭代器失效规则：

- 扩容后：所有迭代器、指针、引用全部失效
- 未扩容的尾部插入：原有元素的迭代器、指针、引用仍有效，但旧 `end()` 失效
- `insert`：插入位置及其之后的迭代器失效；如果触发扩容则全部失效
- `erase`：删除位置及其之后的迭代器失效，删除位置之前仍有效

使用建议：

- 已知数据规模时使用 `reserve()`，减少扩容次数
- 遍历密集、访问密集的场景优先考虑 `vector`
- 不要长期保存 `vector<T>` 内部元素的指针或引用
- 如果需要稳定对象地址，可以考虑 `vector<unique_ptr<T>>`、对象池或其他地址稳定结构

### 1.2 `deque`

`deque` 是双端队列，通常由一组固定大小的连续块组成，再用一个 map 结构管理这些块。

核心复杂度：

| 操作 | 复杂度 | 说明 |
| --- | --- | --- |
| 随机访问 | `O(1)` | 需要先定位块，再定位块内偏移 |
| `push_front()` / `push_back()` | `O(1)` | 头尾扩展高效 |
| `pop_front()` / `pop_back()` | `O(1)` | 头尾删除高效 |
| 中间插入/删除 | `O(n)` | 需要移动元素 |

使用建议：

- 需要频繁头尾插入删除时使用
- 如果主要需求是连续内存和极致遍历性能，`vector` 通常更合适

### 1.3 `list`

`list` 是双向链表，每个节点通常包含：

- 数据
- 前驱指针
- 后继指针

核心复杂度：

| 操作 | 复杂度 | 说明 |
| --- | --- | --- |
| 随机访问 | 不支持 | 只能遍历 |
| 已知位置插入/删除 | `O(1)` | 不需要移动其他元素 |
| 查找 | `O(n)` | 线性遍历 |

迭代器失效规则：

- 插入不会导致已有迭代器失效
- 删除某个节点只会让指向该节点的迭代器失效

使用建议：

- 不要因为“插入删除是 `O(1)`”就默认使用 `list`
- `list` 节点分散在堆上，cache locality 差，实际遍历性能经常弱于 `vector`
- 只有在频繁持有节点迭代器并在已知位置插入删除时，`list` 才更有优势

## 2. 关联容器

| 容器 | 底层实现 | 是否有序 | 查找 | 插入 | 删除 | 典型特点 |
| --- | --- | --- | --- | --- | --- | --- |
| `set` | 红黑树 | 有序 | `O(log n)` | `O(log n)` | `O(log n)` | key 唯一 |
| `multiset` | 红黑树 | 有序 | `O(log n)` | `O(log n)` | `O(log n)` | key 可重复 |
| `map` | 红黑树 | 有序 | `O(log n)` | `O(log n)` | `O(log n)` | key-value，key 唯一 |
| `multimap` | 红黑树 | 有序 | `O(log n)` | `O(log n)` | `O(log n)` | key-value，key 可重复 |

### 2.1 `map`

`map` 通常基于红黑树实现。红黑树是一种自平衡二叉搜索树，通过旋转和染色维持近似平衡。

特点：

- key 按比较函数排序
- 查找、插入、删除都是 `O(log n)`
- 迭代器按 key 的升序遍历
- 插入和删除一般不会导致其他节点的迭代器失效

适合场景：

- 需要有序遍历
- 需要范围查询，例如 `lower_bound()`、`upper_bound()`
- 需要稳定的最坏时间复杂度

订单簿例子：

```cpp
// 买盘价格高优先
std::map<int64_t, Orders, std::greater<int64_t>> bids;

// 卖盘价格低优先
std::map<int64_t, Orders> asks;
```

## 3. 无序关联容器

| 容器 | 底层实现 | 是否有序 | 平均查找 | 平均插入 | 平均删除 | 最坏情况 |
| --- | --- | --- | --- | --- | --- | --- |
| `unordered_set` | 哈希表 | 无序 | `O(1)` | `O(1)` | `O(1)` | `O(n)` |
| `unordered_map` | 哈希表 | 无序 | `O(1)` | `O(1)` | `O(1)` | `O(n)` |
| `unordered_multiset` | 哈希表 | 无序 | `O(1)` | `O(1)` | `O(1)` | `O(n)` |
| `unordered_multimap` | 哈希表 | 无序 | `O(1)` | `O(1)` | `O(1)` | `O(n)` |

### 3.1 `unordered_map`

`unordered_map` 通常基于哈希表实现。元素会根据 `hash(key)` 映射到 bucket 中，发生哈希冲突时使用链表、节点链或其他冲突处理方式。

关键概念：

- `bucket_count()`：bucket 数量
- `load_factor()`：当前负载因子，约等于 `size / bucket_count`
- `max_load_factor()`：允许的最大负载因子
- `rehash()`：重新分配 bucket，并重新放置所有元素
- `reserve(n)`：预留足够 bucket，以容纳大约 n 个元素

复杂度特点：

- 平均查找、插入、删除是 `O(1)`
- 如果哈希冲突严重，最坏可能退化到 `O(n)`
- `rehash` 会导致整体搬迁，代价较高

迭代器失效规则：

- 插入如果触发 rehash，所有迭代器失效
- 插入未触发 rehash，通常已有迭代器仍有效
- 删除只会让被删除元素的迭代器失效
- rehash 后迭代器失效，但指向元素的引用和指针在标准上通常保持有效，除非元素本身被删除

使用建议：

- 延迟敏感场景中提前 `reserve()`
- 自定义 key 时要提供高质量 hash 和相等比较
- 不要依赖遍历顺序
- 如果需要稳定顺序或范围查询，选择 `map`

## 4. 容器适配器

| 容器 | 默认底层容器 | 核心操作 | 复杂度 | 典型用途 |
| --- | --- | --- | --- | --- |
| `stack` | `deque` | `push` / `pop` / `top` | `O(1)` | 后进先出 |
| `queue` | `deque` | `push` / `pop` / `front` / `back` | `O(1)` | 先进先出 |
| `priority_queue` | `vector` + heap | `push` / `pop` / `top` | `O(log n)` / `O(log n)` / `O(1)` | 优先级队列 |

### 4.1 `priority_queue`

`priority_queue` 默认是大顶堆，底层通常使用 `vector` 存储堆。

核心复杂度：

- `top()`：`O(1)`
- `push()`：`O(log n)`
- `pop()`：`O(log n)`

适合场景：

- 每次只需要取最大值或最小值
- 不需要遍历全部有序元素

如果需要频繁删除任意元素，标准 `priority_queue` 不方便，因为它不支持按迭代器删除。

## 5. 特殊容器

| 容器 | 底层实现 | 特点 |
| --- | --- | --- |
| `string` | 连续字符数组 | 类似 `vector<char>`，通常有 small string optimization |
| `bitset<N>` | 固定大小位数组 | 编译期固定大小，每个元素占 1 bit |

### 5.1 `bitset`

`bitset<N>` 是固定大小的二进制位集合，适合保存大量布尔状态。

常用接口：

| 操作 | 含义 |
| --- | --- |
| `count()` | 统计 1 的数量 |
| `any()` | 是否至少有一个 1 |
| `none()` | 是否全部为 0 |
| `set(pos, value)` | 设置某一位 |
| `reset(pos)` | 清零某一位 |
| `flip(pos)` | 翻转某一位 |
| `test(pos)` | 检查某一位是否为 1 |

## 6. 面试高频对比

### 6.1 `vector` vs `list`

`vector`：

- 内存连续，cache locality 好
- 支持 `O(1)` 随机访问
- 中间插入删除需要移动元素
- 扩容会导致迭代器、指针、引用失效

`list`：

- 节点分散，cache locality 差
- 不支持随机访问
- 已知位置插入删除是 `O(1)`
- 插入通常不影响已有迭代器

面试回答重点：

> 如果主要是遍历和随机访问，优先考虑 `vector`。即使 `list` 理论上插入删除是 `O(1)`，实际性能也可能因为 cache miss 和额外内存分配而变差。

### 6.2 `map` vs `unordered_map`

`map`：

- 红黑树
- key 有序
- 查找、插入、删除稳定 `O(log n)`
- 支持范围查询

`unordered_map`：

- 哈希表
- key 无序
- 平均 `O(1)`，最坏 `O(n)`
- 可能 rehash，延迟不稳定

面试回答重点：

> 如果需要有序遍历、范围查询或稳定最坏复杂度，使用 `map`。如果只做 key-value 精确查找，并且能接受 rehash 和哈希冲突风险，可以使用 `unordered_map`，最好提前 `reserve()`。

### 6.3 `push_back` vs `emplace_back`

`push_back`：

- 接收一个已有对象
- 可能发生拷贝或移动

`emplace_back`：

- 直接把构造参数传给元素类型的构造函数
- 可以在容器尾部原地构造对象

示例：

```cpp
std::vector<Order> orders;

orders.push_back(Order{1, 100}); // 先构造临时对象，再移动进 vector
orders.emplace_back(1, 100);     // 直接在 vector 尾部构造
```

注意：如果 `vector` 扩容，已有元素仍然需要移动或拷贝到新内存。

## 7. 量化交易系统中的选择建议

| 场景 | 常见选择 | 原因 |
| --- | --- | --- |
| 行情快照顺序存储 | `vector` | 遍历快，内存连续 |
| 订单 id 到订单对象映射 | `unordered_map` | 精确查找平均 `O(1)` |
| 价格档位排序 | `map` | 买卖盘需要按价格排序 |
| 固定数量状态标记 | `bitset` | 空间紧凑，位运算快 |
| 任务队列 | `queue` / ring buffer | FIFO 语义清晰 |
| Top N 或最优价格候选 | `priority_queue` | 快速获取最大/最小值 |

低延迟系统里的常见原则：

- 能预分配就预分配：`reserve()`、对象池、固定数组
- 避免主路径频繁动态分配
- 避免在热路径触发 `vector` 扩容或 `unordered_map` rehash
- 优先考虑内存连续的数据结构
- 不要长期保存可能失效的迭代器、指针、引用

## 8. 总结速记

- `vector`：连续数组，随机访问快，扩容风险大
- `deque`：分段数组，头尾操作快
- `list`：双向链表，已知位置插入删除快，但遍历慢
- `map`：红黑树，有序，稳定 `O(log n)`
- `unordered_map`：哈希表，平均 `O(1)`，小心 rehash
- `priority_queue`：堆，适合快速取最大/最小
- `bitset`：固定大小位集合，适合大量布尔状态
