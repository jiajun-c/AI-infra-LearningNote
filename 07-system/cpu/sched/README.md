# CPU 调度

CPU 调度解决的是：当多个任务都想使用 CPU 时，操作系统如何决定“谁先运行、运行多久、什么时候切换到别的任务”。

调度器关注的通常不是所有进程，而是处于 `Runnable` 状态的任务。

```text
Runnable：可以运行，只是在等待 CPU
Running ：正在 CPU 上运行
Waiting ：暂时不能运行，正在等待 I/O、锁、定时器、网络事件等
```

## 核心问题

调度器通常在几个目标之间权衡：

```text
公平性：多个 Runnable 任务都能拿到 CPU
响应速度：交互式任务不要被长时间卡住
吞吐量：尽量减少空转和不必要的切换
开销：上下文切换、缓存失效、跨核迁移都不是免费的
```

所以调度不是单纯排队，而是在有限 CPU 核心上不断做选择。

## 为什么需要调度

如果系统中只有一个任务，那么 CPU 可以一直执行它。

但真实系统里会同时有很多任务：

```text
浏览器
终端
数据库
后台服务
内核线程
```

CPU 核心数量有限，所以操作系统需要不断在任务之间切换，让用户感觉多个任务在“同时运行”。

## 时间片和抢占

现代操作系统通常使用抢占式调度。

也就是说，即使一个任务没有主动让出 CPU，操作系统也可以通过定时器中断把它切走。

例如：

```text
Task A running
    |
    | 时间片用完
    v
Task A -> Runnable
调度器选择 Task B
Task B -> Running
```

这里 Task A 并不是不能继续运行，而是它已经使用了一段 CPU 时间。为了公平，调度器把它放回就绪队列，让其他任务也有机会运行。

时间片太大，交互响应会变慢；时间片太小，上下文切换开销会变大。

## Waiting 状态和主动让出 CPU

任务不一定总是因为时间片用完才被切走。

如果一个任务运行过程中需要等待某个事件，比如磁盘 I/O、网络数据、锁、睡眠定时器，它就会进入 `Waiting` 状态。

例如：

```text
Task A running
    |
    | read() 需要等待磁盘数据
    v
Task A -> Waiting
调度器选择 Task B
Task B -> Running
```

这时 Task A 暂时不能继续执行，所以 CPU 不会等待它，而是立刻运行其他 `Runnable` 任务。

等 I/O 完成之后：

```text
磁盘数据准备完成
Task A -> Runnable
之后重新参与调度
```

可以把状态流转简化成：

```text
            时间片到 / 被抢占
Running --------------------> Runnable
   ^                            |
   |                            |
   |        被调度器选中         |
   +----------------------------+

Running ----等待 I/O/锁/事件----> Waiting
Waiting ----事件完成------------> Runnable
```

所以：

```text
时间片到：
任务还能继续运行，但为了公平被切走。

进入 Waiting：
任务暂时不能运行，所以让出 CPU。
```

## CPU 密集型和 I/O 密集型

不同类型任务的调度行为不一样。

CPU 密集型任务主要消耗计算资源，常见状态变化是：

```text
Running -> Runnable -> Running
```

它经常是因为时间片用完而被抢占。

I/O 密集型任务经常等待外部事件，常见状态变化是：

```text
Running -> Waiting -> Runnable -> Running
```

它每次运行一小段时间后就去等待 I/O。I/O 完成后，它重新进入 `Runnable` 队列。

交互式任务通常更像 I/O 密集型任务，比如终端、编辑器、浏览器输入事件。调度器通常会尽量照顾这类任务的响应速度。

## 常见调度策略

### 先来先服务

FCFS，First Come First Served。

谁先进入就绪队列，谁先运行。

优点是简单；缺点是前面的长任务可能阻塞后面的短任务，产生护航效应。

```text
A 需要 10ms
B 需要 1ms
C 需要 1ms
```

如果 A 先运行，B 和 C 都要等待较久。

### 短作业优先

SJF，Shortest Job First。

优先运行预计执行时间最短的任务。

优点是平均等待时间通常较短；缺点是很难准确知道任务还要运行多久，而且长任务可能长期得不到 CPU。

### 最短剩余时间优先

SRTF，Shortest Remaining Time First。

它是短作业优先的抢占式版本。

如果当前任务还剩 8ms，这时来了一个只需要 2ms 的任务，调度器可以暂停当前任务，先运行更短的任务。

优点是短任务响应快；缺点是切换更频繁，长任务也可能饥饿。

### 时间片轮转

RR，Round Robin。

每个任务获得一个固定时间片，时间片用完后放回队尾。

```text
A 运行一个时间片
B 运行一个时间片
C 运行一个时间片
A 继续运行
```

优点是公平，适合交互式系统；缺点是时间片大小需要权衡。

### 优先级调度

每个任务有一个优先级，优先级高的任务优先运行。

优点是重要任务可以优先执行；缺点是低优先级任务可能长期得不到 CPU。

为了解决饥饿，系统可以使用老化机制：等待时间越长，优先级逐渐提高。

### 多级反馈队列

MLFQ，Multi-Level Feedback Queue。

它会根据任务行为动态调整优先级。

基本思想：

```text
新任务先进入高优先级队列
如果总是用完整个时间片，说明偏 CPU 密集型，逐渐降级
如果经常等待 I/O，说明偏交互型，可以保持较高优先级
等待太久的任务可以被提升优先级，避免饥饿
```

这种策略兼顾响应速度和吞吐量，但实现和参数设计更复杂。

## Linux CFS 调度

Linux 普通进程走 `fair` 调度类。传统上常说它使用 CFS，Completely Fair Scheduler。

CFS 不只是简单地给每个任务固定时间片，而是维护每个任务的虚拟运行时间 `vruntime`。

直觉上：

```text
谁运行得越少，vruntime 越小
调度器优先选择 vruntime 最小的任务
```

这样可以让多个可运行任务尽量公平地分享 CPU。

进程的 `nice` 值会影响权重：

```text
nice 越低，权重越高，获得更多 CPU 时间
nice 越高，权重越低，获得更少 CPU 时间
```

更准确地说，`nice` 改变的是任务权重，权重再影响真实运行时间折算成 `vruntime` 的速度：

```text
nice 低 -> 权重大 -> vruntime 增长慢 -> 更容易继续被选中
nice 高 -> 权重小 -> vruntime 增长快 -> 获得的 CPU 时间更少
```

Linux 6.6 之后，`fair` 调度类开始引入 EEVDF。理解时可以先抓住主线：普通任务仍在 `fair` 调度类里竞争 CPU，经典 CFS 用 `vruntime` 表达公平性，新实现还会结合 eligible 和 virtual deadline 改进选择逻辑。

## 动手观察 nice

目录里的 `cfs_demo.cpp` 会创建两个 CPU 密集型子进程，把它们绑到同一个 CPU，然后统计一段时间内各自消耗的 CPU 时间：

```bash
g++ -O2 -std=c++17 cfs_demo.cpp -o cfs_demo
./cfs_demo
```

预期现象：

```text
normal       nice= 0  获得更多 CPU time
low-priority nice=10  获得更少 CPU time
```

如果两个任务没有绑到同一个 CPU，它们可能分别跑在不同核心上，结果就不容易体现调度权重差异。

`cpuset.cpp` 演示的是另一个概念：通过 `sched_setaffinity()` 限制当前进程可以在哪些 CPU 上运行。

```bash
g++ -O2 -std=c++17 cpuset.cpp -o cpu_set
./cpu_set
```

更多命令和源码入口整理在 [tools.md](tools.md)。

## 小结

```text
FCFS：先来先跑
SJF ：短的先跑
RR  ：按时间片轮流跑
优先级：重要的先跑
MLFQ：根据任务行为动态调整
CFS ：尽量公平地分配 CPU 时间
```

CPU 调度的核心不是单纯地“排队”，而是在公平性、响应速度、吞吐量和上下文切换开销之间做权衡。
