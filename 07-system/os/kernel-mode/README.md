# 内核态/用户态

操作系统会把程序执行分成不同权限级别：

```text
用户态：普通应用程序运行的地方，权限低
内核态：操作系统内核运行的地方，权限高
```

用户态程序不能直接操作硬件、修改页表、关闭中断、访问内核内存或读写其他进程的内存。需要系统资源时，它必须通过受控入口进入内核态，由内核检查权限并代为执行。

## 为什么要区分用户态和内核态

核心原因是保护系统。

如果没有权限隔离，任何普通程序都可以直接操作硬件和内核数据结构。一个程序 bug 就可能破坏整个系统，甚至读取其他进程的数据。

区分用户态和内核态后：

```text
应用程序在用户态安全运行
关键资源由内核态统一管理
应用需要资源时，通过系统调用请求内核
```

这样可以带来几个好处：

- 安全隔离：防止应用随意访问内核或其他进程数据。
- 稳定性：应用崩溃通常只影响自己，不会拖垮整个系统。
- 权限控制：文件、网络、设备和进程操作都由内核统一检查。
- 资源管理：CPU、内存、磁盘、网络由内核统一分配和调度。
- 硬件抽象：应用不用直接理解磁盘、网卡、页表等底层细节。

## 用户态

用户态是普通程序运行的地方。

例如：

```text
浏览器
数据库
编辑器
终端程序
业务服务
```

普通计算通常只在用户态完成：

```text
整数加法
数组遍历
字符串处理
函数调用
排序计算
```

这些操作不需要内核介入。

## 内核态

内核态是操作系统内核运行的地方，权限更高。

内核负责管理全局资源：

```text
进程调度
内存管理
文件系统
网络协议栈
设备驱动
中断处理
权限检查
```

这些能力不能直接暴露给普通应用，否则系统就失去了隔离和安全性。

## 什么时候会陷入内核态

从用户态进入内核态，通常叫陷入内核态。

会陷入内核态的操作大体分成三类：

```text
系统调用：程序主动请求内核服务
中断    ：硬件事件打断当前执行
异常/陷阱：CPU 执行指令时发现特殊情况
```

### 系统调用

系统调用是应用主动请求内核帮忙。

常见操作：

```text
文件 I/O：open、read、write、close
网络 I/O：socket、connect、accept、send、recv
进程管理：fork、execve、wait、exit
线程同步：futex，pthread_mutex 竞争时可能触发
内存管理：mmap、munmap、brk、mprotect
设备访问：ioctl
权限信息：chmod、chown、getpid、getuid
时间相关：nanosleep、部分 clock_gettime 场景
```

例如读文件：

```text
用户态程序调用 read()
        |
        v
触发 syscall 指令
        |
        v
CPU 切换到内核态
        |
        v
内核检查 fd、权限、用户缓冲区地址
        |
        v
内核从文件系统/设备读取数据
        |
        v
返回用户态
```

### 中断

中断来自硬件，不一定是当前程序主动触发。

常见中断：

```text
定时器中断：更新时间、检查时间片、触发调度机会
磁盘中断：磁盘 I/O 完成
网卡中断：收到网络包
键盘/鼠标中断：输入事件
设备中断：其他外设完成任务
```

定时器中断是抢占式调度的基础：

```text
用户程序正在运行
        |
        v
定时器中断到来
        |
        v
CPU 进入内核态
        |
        v
内核更新当前任务运行时间
        |
        v
如果时间片用完，可能切换到其他任务
```

### 异常和陷阱

异常/陷阱是 CPU 执行指令时遇到特殊情况。

常见情况：

```text
缺页异常 page fault
除零异常
非法指令
访问无权限内存
断点调试 trap
浮点异常
```

缺页异常很常见。

程序访问的是虚拟地址。如果这个虚拟地址暂时没有对应的物理页，CPU 会触发缺页异常，进入内核态，让内核处理。

```text
用户态访问虚拟地址
        |
        v
页表中没有可用映射
        |
        v
CPU 触发 page fault
        |
        v
进入内核态
        |
        v
内核分配物理页，或从磁盘加载页面
        |
        v
返回用户态继续执行
```

如果这是非法访问，内核无法修复，就会给进程发送 `SIGSEGV`。

## 进入内核态不等于发生调度

进入内核态只是权限模式切换，不一定会切换进程。

例如一次很快的 `getpid()`：

```text
用户态 Task A
        |
        v
内核态执行 getpid
        |
        v
返回用户态 Task A
```

整个过程仍然是同一个任务继续运行。

但如果系统调用需要等待 I/O，或者定时器中断发现时间片用完，就可能发生调度：

```text
Task A 进入内核态
        |
        v
Task A 进入 Waiting，或被抢占
        |
        v
调度器选择 Task B
        |
        v
Task B 运行
```

所以：

```text
进入内核态：CPU 开始执行内核代码
发生调度  ：内核决定换另一个任务运行
```

调度一定发生在内核态中，但进入内核态不一定会调度。

## 小结

```text
用户态：普通应用运行，权限低，不能直接操作关键资源
内核态：操作系统内核运行，权限高，负责管理系统资源

系统调用：应用主动进入内核态
中断    ：硬件事件让 CPU 进入内核态
异常/陷阱：CPU 发现特殊情况后进入内核态
```
